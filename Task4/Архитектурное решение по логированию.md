# Архитектурное решение по логированию

## 1. Контекст системы в разрезе логирования

У "Александрита" уже запланированы метрики и трейсинг, но этого недостаточно для быстрого разбора инцидентов. Метрики из `Task2` показывают, что деградирует массово, а трейсинг из `Task3` показывает, где застрял конкретный заказ. Логирование должно закрыть третий вопрос: что именно произошло внутри сервиса, обработчика очереди или внешней зависимости в момент сбоя.

Для текущей архитектуры это особенно важно, потому что путь заказа проходит через несколько технических границ:

- `B2C`: `Internet Shop -> Shop API -> RabbitMQ -> MES API -> RabbitMQ -> CRM API`;
- `B2B` в переходном состоянии: `API user -> MES API -> RabbitMQ -> CRM API`;
- расчёт стоимости в `MES API` зависит от чтения 3D-модели из `S3`, записи в `MES DB` и публикации результата обратно в `RabbitMQ`.

Проблема не только в самих ошибках, но и в отсутствии операционного контекста:

- нельзя быстро понять, был ли заказ сохранён до публикации события;
- трудно отличить таймаут чтения из `S3` от сбоя записи в БД или ошибки обработки сообщения;
- поддержка и разработка тратят много времени на ручную сверку нескольких систем и таблиц;
- по B2B-потоку особенно болезненно расследовать случаи, когда заказ есть в `MES`, но не дошёл до `CRM`.

Поэтому логирование должно быть не набором произвольных текстовых сообщений, а структурированным контуром расследования инцидентов, связанным с уже выбранными метриками и трейсами.

## 2. Какие системы нужно покрыть логированием

### 2.1. Приоритет первого этапа

Если команда не может внедрить логирование одновременно во все узлы, порядок должен быть таким:

1. `MES API`, `RabbitMQ` и потребительский путь `RabbitMQ -> CRM API`.
2. `Shop API`.
3. `Shop DB`, `MES DB`, `S3` и frontend error logging для `Internet Shop`, `CRM`, `MES`.

Этот порядок повторяет логику `Task1`, `Task2` и `Task3`: сначала нужно сделать наблюдаемой самую рискованную часть пути заказа, где теряются B2B-заказы, зависают расчёты и ломается передача результата в `CRM`.

### 2.2. Почему именно эти системы первыми

- `MES API` совмещает расчёт стоимости, работу с `RabbitMQ`, чтение из `S3`, запись в `MES DB` и переходный B2B-вход.
- `RabbitMQ` является асинхронной границей, на которой сообщение может зависнуть, уйти в retry, оказаться в `dead-letter` или потерять контекст.
- `CRM API` завершает критичный путь заказа: именно здесь заказ становится видим продавцам и backoffice-команде.
- `Shop API` важен как входная точка B2C и будущая единая точка входа B2B, но в первую очередь нужно закрыть уже наблюдаемую бизнес-боль на стороне `MES` и `CRM`.

### 2.3. Что не является первым приоритетом

- Полное клиентское логирование браузерных SPA.
- Детальная запись каждого SQL-запроса с параметрами.
- Сохранение полных payload-ов сообщений из `RabbitMQ`.
- Сохранение request/response body для всех API-методов.

Это либо слишком дорого, либо создаёт повышенные риски утечки данных, либо не даёт быстрого выигрыша в расследовании именно потерянных заказов.

## 3. Принципы логирования

### 3.1. Структурированный формат

Базовый формат для приложений и интеграций - структурированные `JSON`-логи. Такой формат нужен, чтобы:

- искать инциденты по полям, а не по свободному тексту;
- связывать логи с метриками и трейсами;
- строить алерты и простую аналитику по именованным событиям;
- упростить маскирование чувствительных данных на этапе сбора.

### 3.2. Обязательные поля

Во все application и integration logs должны попадать как минимум такие поля:

| Поле | Назначение |
| --- | --- |
| `timestamp` | Время события |
| `level` | Уровень логирования |
| `service` | Источник события |
| `environment` | `dev`, `release`, `prod` |
| `instance` | Инстанс приложения или узла |
| `event_name` | Имя бизнесового или технического события |
| `trace_id` | Связка с трейсингом из `Task3` |
| `span_id` | Связка с конкретным span |
| `request_id` | Корреляция внутри HTTP-сценария |
| `order.id` | Идентификатор заказа |
| `channel` | `b2c`, `b2b`, `crm`, `mes` |
| `partner.id` | Идентификатор B2B-партнёра, если применимо |
| `route` или `operation` | HTTP-маршрут или бизнес-операция |
| `message_id` | Идентификатор сообщения в очереди |
| `correlation_id` | Корреляция producer/consumer |
| `status_from` | Предыдущий статус заказа |
| `status_to` | Новый статус заказа |
| `result` | `success`, `retry`, `failed`, `timeout` |
| `duration_ms` | Длительность операции |
| `error_code` | Код прикладной или инфраструктурной ошибки |

Для инфраструктурных и platform logs набор полей будет зависеть от источника, но `timestamp`, `environment`, `service` или `component`, а также технический `severity` должны быть обязательны.

### 3.3. Корреляция с мониторингом и трейсингом

Логирование должно дополнять уже выбранные контуры наблюдаемости:

- метрики из `Task2` отвечают на вопрос, есть ли массовая деградация;
- трейсы из `Task3` отвечают на вопрос, на каком участке застрял конкретный заказ;
- логи отвечают на вопрос, какое именно событие, исключение или инфраструктурная аномалия произошли внутри узла.

Ключевая связка:

- переход из дашборда в лог-поиск по `service`, `event_name`, `result`;
- переход из лога в `Jaeger` по `trace_id`;
- переход из trace к related logs по `trace_id` и `order.id`.

## 4. Какие логи нужно собирать

### 4.1. Необходимые `INFO`-логи

Ниже список событий, которые должны логироваться на уровне `INFO`, потому что они отражают штатный ход обработки заказа и нужны для расследования инцидентов без включения `DEBUG`.

| Система | `event_name` | Что логировать на `INFO` | Обязательные поля |
| --- | --- | --- | --- |
| `Shop API` | `order.created` | Заказ создан в системе магазина | `order.id`, `channel`, `trace_id`, `request_id`, `result` |
| `Shop API` | `file.upload.completed` | 3D-файл успешно загружен в `S3` | `order.id`, `file.id`, `trace_id`, `duration_ms`, `result` |
| `Shop API` | `order.submitted` | Пользователь подтвердил оформление заказа | `order.id`, `channel`, `trace_id`, `status_to=SUBMITTED` |
| `Shop API` | `order.persisted` | Заказ сохранён в `Shop DB` | `order.id`, `trace_id`, `duration_ms`, `result` |
| `Shop API` | `order.event.published` | Событие о заказе опубликовано в `RabbitMQ` | `order.id`, `trace_id`, `message_id`, `correlation_id`, `result` |
| `Shop API` | `request.completed` | Завершён критичный HTTP-запрос | `route`, `method`, `status_code`, `duration_ms`, `trace_id` |
| `MES API` | `b2b.order.received` | Получен B2B-заказ через переходный API-контур | `order.id`, `partner.id`, `trace_id`, `request_id`, `result` |
| `MES API` | `order.event.consumed` | Получено сообщение о заказе из `RabbitMQ` | `order.id`, `trace_id`, `message_id`, `correlation_id`, `consumer`, `result` |
| `MES API` | `calculation.started` | Начат расчёт стоимости изделия | `order.id`, `trace_id`, `calculation.type`, `result=started` |
| `MES API` | `s3.object.read` | Успешно прочитана 3D-модель | `order.id`, `file.id`, `trace_id`, `duration_ms`, `result` |
| `MES API` | `calculation.completed` | Расчёт стоимости завершён | `order.id`, `trace_id`, `duration_ms`, `result` |
| `MES API` | `calculation.result.persisted` | Результат сохранён в `MES DB` | `order.id`, `trace_id`, `duration_ms`, `result` |
| `MES API` | `price.event.published` | Результат расчёта опубликован в `RabbitMQ` | `order.id`, `trace_id`, `message_id`, `correlation_id`, `result` |
| `MES API` | `status.changed` | Заказ переведён в один из MES-статусов | `order.id`, `status_from`, `status_to`, `trace_id`, `result` |
| `CRM API` | `price.event.consumed` | Получено сообщение с ценой или статусом | `order.id`, `trace_id`, `message_id`, `correlation_id`, `consumer`, `result` |
| `CRM API` | `order.synced` | Заказ или цена успешно применены в `CRM` | `order.id`, `trace_id`, `duration_ms`, `result` |
| `CRM API` | `status.changed` | Продавец или backend перевёл заказ в новый статус | `order.id`, `status_from`, `status_to`, `trace_id`, `result` |
| `CRM API` | `manual.status.override` | Выполнено ручное изменение статуса | `order.id`, `status_from`, `status_to`, `actor_role`, `trace_id`, `result` |
| `CRM API` | `order.closed` | Заказ закрыт | `order.id`, `trace_id`, `status_to=CLOSED`, `result` |
| `RabbitMQ` | `message.accepted` | Сообщение принято брокером | `queue`, `exchange`, `routing_key`, `message_id`, `correlation_id` |
| `RabbitMQ` | `message.delivered` | Сообщение выдано потребителю | `queue`, `consumer`, `message_id`, `correlation_id`, `result` |
| `RabbitMQ` | `message.acked` | Потребитель подтвердил обработку | `queue`, `consumer`, `message_id`, `correlation_id`, `result` |
| `Shop DB`, `MES DB` | `db.connection.opened` | Подключение пула или сессии установлено штатно | `db`, `service`, `instance`, `result` |
| `Shop DB`, `MES DB` | `db.migration.applied` | Прикладная миграция или штатное изменение схемы выполнено | `db`, `service`, `migration_id`, `result` |
| `S3` | `object.put` | Файл успешно записан в хранилище | `bucket`, `file.id`, `order.id`, `result` |
| `S3` | `object.get` | Файл успешно прочитан из хранилища | `bucket`, `file.id`, `order.id`, `result` |

Для `RabbitMQ`, `PostgreSQL` и `S3` в расследовании инцидентов самые полезные события часто рождаются не в самих managed logs, а в прикладочных логах producer/consumer и вызывающего сервиса. Поэтому ставку нужно делать именно на event logs приложений, а platform logs подключать как дополнительный контекст.

### 4.2. Audit logs

Кроме `INFO`-событий по обработке заказа нужен отдельный поток audit logs для ручных действий пользователей внутренних систем:

- кто создал или изменил заказ;
- кто сменил статус;
- кто выполнил ручной override;
- кто закрыл заказ вручную;
- кто инициировал повторную обработку или компенсационную операцию.

Для audit logs обязательны:

- `actor.id` в безопасной внутренней форме;
- `actor.role`;
- `action`;
- `resource_type`;
- `resource_id`;
- `timestamp`;
- `status_from`;
- `status_to`;
- `result`.

## 5. Уровни логирования

### 5.1. `INFO`

Используется для штатных, но операционно значимых событий:

- создание заказа;
- публикация и потребление сообщений;
- смена статуса;
- начало и завершение расчёта;
- завершение критичных HTTP-запросов;
- ручные действия пользователей внутренних систем.

### 5.2. `WARN`

Используется для ситуаций, которые ещё не означают отказ, но требуют внимания:

- повторная доставка сообщения;
- превышение ожидаемой длительности операции;
- fallback-сценарий;
- частично успешная обработка;
- аномальный рост валидационных ошибок;
- временная недоступность внешней зависимости с успешным retry.

Примеры событий на `WARN`:

- `message.retried`;
- `consumer.processing.delayed`;
- `s3.read.slow`;
- `db.query.slow`;
- `request.timeout.recovered`.

### 5.3. `ERROR`

Используется для сбоев, которые привели или почти привели к потере бизнес-операции:

- не удалось сохранить заказ;
- не удалось опубликовать сообщение в `RabbitMQ`;
- расчёт стоимости завершился ошибкой;
- результат не записался в БД;
- consumer получил сообщение, но не смог завершить обработку;
- заказ не был синхронизирован в `CRM`.

Примеры событий на `ERROR`:

- `order.persist.failed`;
- `message.publish.failed`;
- `calculation.failed`;
- `crm.sync.failed`;
- `db.write.failed`;
- `message.dead_lettered`.

### 5.4. `DEBUG`

`DEBUG` не должен быть постоянным режимом в `prod`. Его стоит включать:

- временно при расследовании сложного инцидента;
- точечно для конкретного сервиса или обработчика;
- на ограниченное время в `release` или `prod`.

В `DEBUG` можно писать технические шаги внутри функции или пайплайна, но без payload-ов сообщений, персональных данных, токенов и содержимого 3D-файлов.

## 6. Мотивация

Логирование нужно, чтобы перестать разбирать сбои по словам клиента и вручную сопоставлять поведение нескольких систем. Сейчас у команды нет удобного способа быстро ответить на вопросы:

- был ли заказ сохранён до публикации события;
- дошло ли сообщение до потребителя и подтвердил ли он обработку;
- из-за чего конкретно не завершился расчёт: `S3`, БД, очередь или код сервиса;
- кто и когда вручную поменял статус заказа.

Что это даст компании:

- ускорит расследование инцидентов и сократит нагрузку на поддержку;
- упростит локализацию потери заказа на конкретном техническом шаге;
- даст операционный журнал по ручным действиям продавцов и операторов;
- позволит быстрее отличать единичный сбой от массовой деградации;
- сделает B2B-направление безопаснее с точки зрения SLA и доверия партнёров.

### 6.1. На какие метрики повлияет внедрение логирования

Технические и бизнесовые показатели, которые должны улучшиться:

1. `MTTR` по инцидентам с потерянными или зависшими заказами.
2. Среднее время разбора обращения поддержки по конкретному заказу.
3. Доля инцидентов, для которых удаётся локализовать точку отказа без ручной сверки БД и очередей.
4. Количество повторных обращений по одному и тому же заказу.
5. Доля B2B-заказов, по которым можно быстро восстановить цепочку событий в расследовании.

### 6.2. Для каких систем логирование и трейсинг нужны в первую очередь

В первую очередь нужно настраивать и логирование, и трейсинг для:

- `MES API`;
- `RabbitMQ`;
- `CRM API` на consumer path;
- затем `Shop API`.

Причина одна и та же: именно этот участок даёт максимальную бизнесовую боль и совпадает с критическим путём, выделенным в предыдущих заданиях.

## 7. Предлагаемое решение

Схема решения приведена в файле `Task4/logging_architecture.puml`.

### 7.1. Выбранный стек

Для первого этапа предлагаю такой контур:

- приложения пишут структурированные `JSON`-логи;
- на виртуальных машинах или узлах работает агент сбора `Fluent Bit`;
- `Fluent Bit` читает application logs, platform logs и audit logs, маскирует чувствительные поля и отправляет записи в `Grafana Loki`;
- поиск, фильтрация и оперативный анализ логов выполняются через `Grafana Explore`;
- для перехода к распределённому trace используется `trace_id` и интеграция с `Jaeger`.

Почему именно такой вариант:

- он хорошо сочетается со стеком `Prometheus/Grafana/Alertmanager` из `Task2`;
- не требует отдельной тяжёлой поисковой платформы на первом этапе;
- даёт быстрый путь от дашборда и алерта к нужным логам и trace;
- подходит небольшой команде сопровождения;
- `Loki` индексирует метки, а не полный текст логов, поэтому хорошо подходит для структурированных событий и не требует строить дорогой full-text контур с первого дня.

### 7.2. Какие компоненты нужно внедрить или доработать

1. Доработать `Shop API`, `MES API`, `CRM API`:
   - перейти на structured logging;
   - унифицировать `event_name`;
   - прокидывать `trace_id`, `span_id`, `request_id`, `order.id`, `message_id`, `correlation_id`;
   - разделить application logs и audit logs.
2. Настроить сбор логов `RabbitMQ`, `PostgreSQL` и `S3`:
   - broker logs;
   - consumer/producer integration logs из приложений;
   - slow query и error logs баз;
   - access logs по операциям с объектами в `S3`, если это позволяет используемый контур хранения.
3. Развернуть `Fluent Bit` как единый слой доставки логов.
4. Развернуть `Grafana Loki` как backend хранения логов.
5. Настроить `Grafana` dashboards/explore views для:
   - расследования по `order.id`;
   - расследования по `trace_id`;
   - расследования по `partner.id`;
   - расследования по `event_name` и `result`.

### 7.3. Политика безопасности

В логах не должны храниться в открытом виде:

- персональные данные клиента;
- email, телефон, адрес доставки и комментарии к заказу;
- токены, cookie, API keys, заголовки авторизации;
- содержимое 3D-файлов;
- полные payload-ы сообщений `RabbitMQ`;
- SQL-запросы с параметрами.

Политика доступа:

- `platform/ops` - полный доступ к operational logs и настройкам пайплайна;
- `backend` - доступ к application logs и связанным trace;
- `support-readonly` - read-only доступ к поиску по `order.id`, `trace_id`, `partner.id`, `event_name`;
- внешним клиентам и B2B-партнёрам прямой доступ к логам не предоставляется.

Дополнительно:

- маскирование чувствительных полей должно происходить до записи в backend;
- доступ к `Grafana` и `Loki` должен быть через корпоративную учётную запись;
- действия администраторов и доступ к логам тоже должны логироваться как audit trail.

### 7.4. Политика хранения

Для `Loki` я не предлагаю делать отдельный физический индекс на каждую систему. В отличие от `ELK`-подобных решений, `Loki` индексирует метки и хранит тело логов в chunks, поэтому на первом этапе разумнее разделять данные по потокам и retention policy, а не по отдельным индексам под каждый сервис.

Базовое разделение потоков:

- `application` logs;
- `integration` logs;
- `audit` logs;
- `platform` logs.

Основные метки:

- `environment`;
- `service`;
- `log_type`;
- `channel`;
- `severity`.

Предлагаемый retention:

- `prod` application и integration logs - 30 дней;
- `prod` audit logs - 90 дней;
- `release` logs - 14 дней;
- `dev` logs - 7 дней;
- временно включённый `DEBUG` в `prod` - отдельный короткий retention, не более 3 дней.

Точный объём хранения нельзя честно задать в гигабайтах без реального измерения потока логов. Поэтому sizing должен рассчитываться по фактическому ingestion после запуска:

`average_daily_ingest * retention_days * replication_or_reserve_factor`

Для первого этапа нужно:

- включить замер суточного ingestion по каждому типу потока;
- через 1-2 недели продовых данных зафиксировать storage budget;
- отдельно ограничить `DEBUG` и высокошумные технические логи квотами и коротким retention.

### 7.5. Почему не выбираю более тяжёлую платформу на старте

Полноценные `ELK/OpenSearch/Splunk`-решения дают сильный full-text поиск и более сложную аналитику, но для текущего состояния "Александрита" у них есть минусы:

- выше операционная стоимость и сложность сопровождения;
- больше риск уйти в платформенную интеграцию вместо быстрого закрытия бизнесовой боли;
- для небольшой команды выгоднее переиспользовать уже выбранную `Grafana` как единый интерфейс observability.

## 8. Как превратить сбор логов в анализ логов

### 8.1. Алертинг по логам

Логи не должны оставаться только средством ручного поиска. Для первого этапа нужен rule-based alerting поверх log queries:

1. Алерт на рост `ERROR`-событий в `MES API`, `CRM API`, `Shop API`.
2. Алерт на `message.dead_lettered` или резкий рост `message.publish.failed`.
3. Алерт на всплеск retry-событий по очередям.
4. Алерт на массовый рост `request.timeout.recovered` и других `WARN`-событий деградации.
5. Алерт на аномальный рост `b2b.order.received` или `order.created`, если это выходит за обычный профиль нагрузки и похоже на некорректную интеграцию или атаку.

Если используется единая `Grafana`, эти правила можно строить как log-based alerts рядом с уже существующими метриками и trace-derived metrics.

### 8.2. Поиск аномалий

На первом этапе не нужен сложный ML-контур. Достаточно базовых сценариев поиска аномалий:

- резкий рост частоты одинаковых ошибок;
- всплеск невалидных запросов от одного `partner.id`;
- аномальный рост числа повторных доставок сообщений;
- рост slow query events;
- резкое изменение соотношения `INFO` к `ERROR` по критическому сервису.

Такой подход проще внедрить и поддерживать, а пользы для текущей команды он даст больше, чем дорогая аналитическая платформа.

## 9. Критерии выбора технологии для работы с логами

Дополнительное задание разумно закрыть сравнением четырёх вариантов: `Grafana Loki`, `OpenSearch`, `ELK`, `Splunk`.

Для текущего состояния "Александрита" базовый выбор - `Grafana Loki`, потому что он лучше соответствует уже выбранной observability-архитектуре и даёт минимальный путь к пользе без избыточной платформенной сложности.

## 10. Итог

Для "Александрита" логирование должно стать не просто сбором ошибок, а операционным журналом прохождения заказа:

- на входе в `Shop API` и `MES API`;
- на публикации и потреблении сообщений в `RabbitMQ`;
- в расчёте стоимости в `MES API`;
- при применении результата в `CRM API`;
- на ручных действиях продавцов и операторов.

Если внедрить structured logs, корреляцию с `trace_id`, логический разбор потоков и базовый alerting по логам, команда перестанет расследовать инциденты вслепую. В связке с мониторингом из `Task2` и трейсингом из `Task3` это даст полноценный минимальный observability-контур для поиска причин потери и зависания заказов.

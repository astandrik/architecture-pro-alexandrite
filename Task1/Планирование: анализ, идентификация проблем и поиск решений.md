# Планирование: анализ, идентификация проблем и поиск решений

## 1. Существующие и потенциальные проблемные места

### Существующие проблемы

1. **MES API перегружен.** В одном сервисе собраны расчёт стоимости (2–30 мин), B2B API, дашборд операторов, статусы и работа с очередью. При росте нагрузки сервис начинает сам себе мешать - api и калькулятор стоимости в одном приложении. Если единственный инстанс падает, команда может узнать об этом не сразу: алертов нет.

2. **Заказы теряются, но непонятно где.** Нет трейсинга и мониторинга. Заказ проходит через несколько систем (Shop API → RabbitMQ → MES API → RabbitMQ → CRM API) и на любом этапе может застрять. Без диагностики причина неизвестна.

3. **B2B-сценарий особенно хрупкий.** Партнёр шлёт заказ в MES API, оттуда через RabbitMQ он должен попасть в CRM. Если не попал — заказ есть только в MES db, продавец его не видит.

4. **Дашборд операторов тормозит.** Операторы теряют деньги — кто первый взял заказ, тот и получит оплату. MES API одновременно считает модели и отдаёт данные из той же MES db — конкуренция за ресурсы. Фильтр и пагинация не помогли.

5. **Всё одноинстансовое и на виртуалках.** Каждый сервис и каждая база работают в одном экземпляре на отдельной виртуалке. Виртуалка может просто упасть, а команда заметит это не сразу. Отказ одного ДЦ — нормальная ситуация для облака, но при текущей схеме это остановит часть бизнеса.

6. **Медленный релизный контур.** Один QA, ручной E2E, ручной деплой в `release` и `production`. Исправления долго доходят до продакшена, а баги живут дольше, чем хотелось бы.

### Потенциальные проблемы

7. **CRM API и Shop API делят одну Shop DB.** При росте нагрузки будут конкурировать за один PostgreSQL-инстанс.

8. **Рост backlog на расчёте стоимости.** +100 заказов/месяц + B2B, один инстанс MES — будет только хуже.

9. **Полная остановка при отказе узла.** При одноинстансовой архитектуре — вопрос времени.

## 2. Инициативы в порядке приоритета

Ключевой принцип плана на ближайшие полгода — не делать `big bang`-перестройку уже работающей продовой системы, а совмещать быстрые выигрыши на текущей архитектуре с постепенным переносом ответственности в целевые контуры. По сути это `strangler fig`: сначала стабилизируем и делаем систему наблюдаемой, затем поэтапно выносим B2B-вход, operator dashboard и управление статусами из перегруженного `MES API`, не отключая старый сценарий одним шагом.

### 1. Настроить метрики, алерты, трейсинг, дашборды и защитные механизмы на текущей инфраструктуре

Сейчас система уже работает в продовом окружении, но команда почти не видит, где именно завис заказ, какая очередь растёт, почему тормозит `MES API` и не упёрлась ли база в соединения. Из-за этого любые следующие шаги, включая платформенные миграции, придётся делать почти вслепую.

Первым шагом должен стать базовый контур наблюдаемости и минимальные защитные меры на текущих виртуалках:

- health-check и алерты на недоступность сервисов;
- метрики по сервисам (`RPS`, `latency`, ошибки, `CPU`, память);
- метрики по `RabbitMQ` (`queue depth`, `dead letters`, `in flight`, age of oldest message);
- метрики по базам (соединения, slow queries, нагрузка);
- distributed tracing по критическому пути заказа (`Shop API -> RabbitMQ -> MES API -> RabbitMQ -> CRM API`);
- дашборды для команды и саппорта;
- backpressure для `MES API`: ограничение параллельных расчётов, лимиты на `in-flight` обработку сообщений из `RabbitMQ`, защита operator dashboard от конкуренции с тяжёлым расчётом;
- circuit breaker и жёсткие timeout-ы на медленных внешних зависимостях, в первую очередь на чтении 3D-моделей из `S3`, чтобы зависшая зависимость не удерживала поток обработки.

Это самый быстрый и наименее дорогой способ снизить риск для продового окружения и начать принимать дальнейшие архитектурные решения на основе данных, а не ощущений. Важно, что здесь backpressure и circuit breaker не заменяют архитектурное упрощение системы, а дают быстрые защитные меры до переезда в Kubernetes и до выделения `MES` в чистый расчётный контур.

### 2. Перевести B2B-партнёров на Shop API

Сейчас партнёры шлют заказы напрямую в MES API. Это хрупкий путь: заказ создаётся в MES db, и дальше через RabbitMQ должен попасть в CRM. Если сообщение не дошло — заказ потерян.

Решение: перевести партнёров на `Shop API` поэтапно, без резкого отключения текущего пути. `Shop API` уже принимает B2C-заказы, а для B2B нужно только адаптировать авторизацию, например использовать API-ключи вместо пользовательских сессий.

Что это даёт:

- единая точка входа для всех заказов — и B2C, и B2B идут через Shop API;
- B2B-заказ сразу попадает в Shop DB — не зависит от хрупкого пути через RabbitMQ;
- MES API разгружается от внешнего трафика;
- Shop API масштабируется горизонтально в Kubernetes — единая точка входа, но не единая точка отказа.

### 3. Изолировать MES как калькулятор и распараллелить

После перевода B2B-партнёров на `Shop API` `MES API` больше не принимает внешние заказы. Его роль сводится к расчёту стоимости.

- MES остаётся чёрным ящиком для расчёта: берёт 3D-модель из S3 и возвращает цену. Это минимальный объём изменений вокруг чужого кода.
- Такое разделение позволит безопасно масштабировать калькулятор стоимости и снять узкое место в расчётном контуре.
- Дашборд операторов и управление статусами постепенно переезжают в `CRM API` / `Shop API` поэтапно, без единовременного отключения старого сценария - паттерн strangler fig.

### 4. Переезд на Managed Kubernetes

Будет эффективно рассмотреть `Managed Kubernetes` как следующий этап развития платформы. Сам по себе Kubernetes не объяснит, где теряются заказы и что именно тормозит. Если сначала внедрить метрики, алерты и трейсинг, миграция станет заметно безопаснее и управляемее.

Что даст этот этап:

- отказоустойчивость — replicas: `N` вместо одного инстанса;
- более предсказуемый rollout и rollback;
- удобное горизонтальное масштабирование;
- стандартную платформу для дальнейшего роста.

На этом этапе нужно контейнеризировать приложения, поднять кластер, перенести контур наблюдаемости, написать манифесты и настроить CI/CD для деплоя в кластер. `RabbitMQ` можно сразу вынести в управляемый сервис Yandex Cloud, чтобы не держать его на собственных виртуалках.

### 5. Исследовать метрики и найти причины тормозов

На основе метрик и трейсов можно будет проверить:

- почему тормозит дашборд операторов (тяжёлые запросы? конкуренция за MES db? проблема на фронтенде?);
- где именно теряются или застревают заказы;
- где узкие места в обмене сообщениями через RabbitMQ.

Это позволит точечно исправлять реальные узкие места вместо догадок. Часть проблем может исчезнуть уже после выполнения первых четырёх инициатив.

### 6. Автоматизировать E2E-тестирование

Сейчас один QA вручную прогоняет E2E на каждый релиз. Это дорого, медленно и ненадёжно. Автоматизированные E2E-тесты дадут быструю обратную связь при изменениях, особенно перед миграциями вроде перевода B2B на `Shop API` и поэтапного выделения `MES` в отдельный расчётный контур. Эту работу фронтенд-разработчики могут вести параллельно с бэкенд-инициативами.

### 7. Оценить необходимость разделения Shop DB

CRM API и Shop API работают с одной базой. Пока прямых жалоб на это нет, но при росте нагрузки они могут начать конкурировать за один PostgreSQL-инстанс. Решение о разделении базы стоит принимать после появления метрик реальной нагрузки на `Shop DB`. Если разделение понадобится, можно рассмотреть `YDB` как управляемую базу в Yandex Cloud, которая масштабируется горизонтально и не требует ручного шардирования.

## 3. Целевая архитектура через полгода

Целевая C4 Container-диаграмма приведена в файлах `Task1/target_c4_containers.puml` и `Task1/target_c4_containers.png`.

Через полгода картина должна быть такой:

1. Основные сервисы работают в Managed Kubernetes в Yandex Cloud, а не на отдельных виртуалках. Это позволяет масштабировать их по нагрузке, автоматически перезапускать упавшие инстансы и быстрее выкатывать изменения через CI/CD.

2. В системе есть метрики, алерты, distributed tracing и дашборды. По конкретному заказу можно проследить весь путь и быстро понять, на каком участке возникла проблема.

3. Shop API становится единой точкой входа для всех заказов. B2C-клиенты и B2B-партнёры отправляют заказ в один сервис, который сохраняет его в Shop DB и публикует событие на расчёт.

4. MES больше не пытается делать всё сразу и остаётся калькулятором. Он получает задание через RabbitMQ, читает 3D-модель из S3, считает стоимость, сохраняет результат в MES DB и публикует его обратно в очередь.

5. RabbitMQ работает как управляемый сервис в облаке. Он отвечает за асинхронный обмен событиями, но его больше не нужно поддерживать на отдельных виртуалках.

6. В CRM-контур переезжают пользовательские сценарии продавцов и операторов. Текущий CRM UI расширяется под backoffice-задачи, а CRM API управляет заказами, статусами и принимает результаты расчёта из очереди.

7. Shop DB пока остаётся общей для Shop API и CRM API, но это временное решение. Вопрос о разделении базы или переезде на YDB решается позже по реальным метрикам, а не заранее.

В итоге зоны ответственности становятся понятнее: Shop принимает заказ, MES считает стоимость, CRM ведёт операционную работу, RabbitMQ передаёт события, а контур наблюдаемости помогает быстро находить проблемы.

## 4. Если бы в ближайшие полгода можно было сделать только три инициативы

При таком ограничении приоритет был бы таким:

1. **Настроить базовый monitoring, alerting, tracing и дашборды на текущей инфраструктуре.** Это самый быстрый способ снизить риск для уже работающей системы. Сейчас команда не видит, где именно теряются или зависают заказы, поэтому без контура наблюдаемости любые следующие изменения будут делаться почти вслепую.

2. **Перевести B2B-партнёров на Shop API.** Это самый полезный шаг с точки зрения бизнеса. Он убирает самый хрупкий сценарий, делает `Shop API` единой точкой входа и разгружает `MES`. После этого путь заказа становится короче и надёжнее.

3. **Изолировать MES как калькулятор.** Это прямой способ убрать конфликт между расчётом стоимости, B2B API и операторским дашбордом. Такое разделение позволит масштабировать `MES` как расчётный контур, а не как перегруженный комбайн.

Почему не остальные:

- **Managed Kubernetes** важен как платформенный шаг, но сам по себе не снимает главную текущую проблему: без метрик и трейсов команда всё ещё не видит, где теряются заказы и что именно тормозит. Делать такую миграцию без контура наблюдаемости рискованно и дороже.
- **Автоматизация E2E-тестов** полезна, но не снимает главные архитектурные риски быстрее, чем первые три пункта.
- **Исследование тормозов и разделения Shop DB** имеет смысл проводить на основе метрик фактической нагрузки. Иначе решение снова будет строиться на догадках.

# Планирование: анализ, идентификация проблем и поиск решений

## 1. Существующие и потенциальные проблемные места

### Существующие проблемы

1. **MES API перегружен.** В одном сервисе собраны расчёт стоимости (2–30 мин), B2B API, дашборд операторов, статусы и работа с очередью. При росте нагрузки сервис начинает сам себе мешать - api и калькулятор стоимости в одном приложении. Если единственный инстанс падает, команда может узнать об этом не сразу: алертов нет.

2. **Заказы теряются, но непонятно где.** Нет трейсинга и мониторинга. Заказ проходит через несколько систем (Shop API → RabbitMQ → MES API → RabbitMQ → CRM API) и на любом этапе может застрять. Без диагностики причина неизвестна.

3. **B2B-сценарий особенно хрупкий.** Партнёр шлёт заказ в MES API, оттуда через RabbitMQ он должен попасть в CRM. Если не попал — заказ есть только в MES db, продавец его не видит.

4. **Дашборд операторов тормозит.** Операторы теряют деньги — кто первый взял заказ, тот и получит оплату. MES API одновременно считает модели и отдаёт данные из той же MES db — конкуренция за ресурсы. Фильтр и пагинация не помогли.

5. **Всё одноинстансовое и на виртуалках.** Каждый сервис и каждая база работают в одном экземпляре на отдельной виртуалке. Виртуалка может просто упасть, а команда заметит это не сразу. Отказ одного ДЦ — нормальная ситуация для облака, но при текущей схеме это остановит часть бизнеса.

6. **Медленный релизный контур.** Один QA, ручной E2E, ручной деплой в release и prod. Исправления долго доходят до продакшена, а баги живут дольше, чем хотелось бы.

### Потенциальные проблемы

7. **CRM API и Shop API делят одну Shop DB.** При росте нагрузки будут конкурировать за один PostgreSQL-инстанс.

8. **Рост backlog на расчёте стоимости.** +100 заказов/месяц + B2B, один инстанс MES — будет только хуже.

9. **Полная остановка при отказе узла.** При одноинстансовой архитектуре — вопрос времени.

## 2. Инициативы в порядке приоритета

### 1. Переезд на Managed Kubernetes

Сейчас все сервисы крутятся на отдельных виртуалках. Переезд на Managed Kubernetes в Yandex Cloud снимает сразу несколько рисков:

- масштабирование — replicas: N вместо одного инстанса;
- наблюдаемость — логи, метрики, health checks из коробки;
- отказоустойчивость — Kubernetes перезапускает упавшие поды;
- быстрый деплой — rolling update, откат одной командой;
- основа для всех следующих инициатив.

Нужно контейнеризировать приложения, поднять кластер, написать манифесты и настроить CI/CD для деплоя в кластер. RabbitMQ можно сразу вынести в Managed Service в Yandex Cloud, чтобы не держать его на своих виртуалках.

### 2. Настроить метрики, алерты, трейсинг и дашборды

Сразу после переезда в Kubernetes нужно настроить наблюдаемость, пока сама архитектура ещё не менялась:

- метрики по сервисам (RPS, latency, ошибки, CPU/память);
- метрики по RabbitMQ (queue depth, dead letters, consumer lag);
- метрики по базам (нагрузка, slow queries, соединения);
- distributed tracing — сквозной trace id по всему пути заказа (Shop API → RabbitMQ → MES API → RabbitMQ → CRM API), чтобы можно было взять конкретный заказ и увидеть, где он застрял или потерялся;
- алерты на критичные ситуации (сервис не отвечает, очередь растёт, ошибки выше порога);
- дашборды для команды — чтобы видеть состояние системы в реальном времени.

Без этого следующие шаги всё равно будут строиться на догадках.

### 3. Перевести B2B-партнёров на Shop API

Сейчас партнёры шлют заказы напрямую в MES API. Это хрупкий путь: заказ создаётся в MES db, и дальше через RabbitMQ должен попасть в CRM. Если сообщение не дошло — заказ потерян.

Решение: перевести партнёров на Shop API по схеме strangler fig. Shop API уже принимает B2C-заказы, а для B2B нужно только адаптировать авторизацию, например использовать API-ключи вместо пользовательских сессий.

Что это даёт:

- единая точка входа для всех заказов — и B2C, и B2B идут через Shop API;
- B2B-заказ сразу попадает в Shop DB — не зависит от хрупкого пути через RabbitMQ;
- MES API разгружается от внешнего трафика;
- Shop API масштабируется горизонтально в Kubernetes — единая точка входа, но не единая точка отказа.

### 4. Изолировать MES как калькулятор и распараллелить

После инициативы 3 MES API больше не принимает внешние заказы. Дальше его нужно оставить только для расчёта стоимости.

- MES остаётся чёрным ящиком для расчёта: берёт 3D-модель из S3 и возвращает цену. Это минимальный объём изменений вокруг чужого кода.
- В Kubernetes запускаются несколько реплик "калькулятора стоимости" в mes. Это самый прямой способ снять узкое место после инициативы 1.
- Дашборд операторов и управление статусами постепенно переезжают в CRM API / Shop API по схеме strangler fig.

### 5. Исследовать метрики и найти причины тормозов

После переезда в Kubernetes и изоляции MES-калькулятора у нас появятся логи, метрики и трейсы. Дальше уже по данным нужно проверить:

- почему тормозит дашборд операторов (тяжёлые запросы? конкуренция за MES db? проблема на фронтенде?);
- где именно теряются или застревают заказы;
- где узкие места в обмене сообщениями через RabbitMQ.

После этого можно точечно чинить реальные узкие места, а не гадать. Часть проблем к этому моменту может исчезнуть сама после инициатив 1–4.

### 6. Автоматизировать E2E-тестирование

Сейчас один QA вручную прогоняет E2E на каждый релиз. Это дорого, медленно и ненадёжно. Автоматизированные E2E-тесты дадут быструю обратную связь при изменениях, особенно перед миграциями вроде B2B на Shop API и strangler fig для MES. Эту работу фронтенд-разработчики могут вести параллельно с бэкенд-инициативами.

### 7. Оценить необходимость разделения Shop DB

CRM API и Shop API работают с одной базой. Пока прямых жалоб на это нет, но при росте нагрузки они могут начать конкурировать за один PostgreSQL-инстанс. После появления метрик нужно посмотреть на реальную нагрузку на Shop DB и решить, нужно ли разделять базу. Если да, можно рассмотреть YDB как managed-базу в Yandex Cloud, которая масштабируется горизонтально и не требует ручного шардирования.

## 3. Целевая архитектура через полгода

Целевая C4 Container-диаграмма приведена в файлах `Task1/target_c4_containers.puml` и `Task1/target_c4_containers.png`.

Через полгода картина должна быть такой:

1. Основные сервисы работают в Managed Kubernetes в Yandex Cloud, а не на отдельных виртуалках. Это позволяет масштабировать их по нагрузке, автоматически перезапускать упавшие инстансы и быстрее выкатывать изменения через CI/CD.

2. В системе есть метрики, алерты, distributed tracing и дашборды. По конкретному заказу можно проследить весь путь и быстро понять, на каком участке возникла проблема.

3. Shop API становится единой точкой входа для всех заказов. B2C-клиенты и B2B-партнёры отправляют заказ в один сервис, который сохраняет его в Shop DB и публикует событие на расчёт.

4. MES больше не пытается делать всё сразу и остаётся калькулятором. Он получает задание через RabbitMQ, читает 3D-модель из S3, считает стоимость, сохраняет результат в MES DB и публикует его обратно в очередь.

5. RabbitMQ работает как управляемый сервис в облаке. Он по-прежнему отвечает за асинхронный обмен событиями, но его не нужно поддерживать на отдельных виртуалках.

6. В CRM-контур переезжают пользовательские сценарии продавцов и операторов. Текущий CRM UI расширяется под backoffice-задачи, а CRM API управляет заказами, статусами и принимает результаты расчёта из очереди.

7. Shop DB пока остаётся общей для Shop API и CRM API, но это временное решение. Вопрос о разделении базы или переезде на YDB решается позже по реальным метрикам, а не заранее.

В итоге зоны ответственности становятся понятнее: Shop принимает заказ, MES считает стоимость, CRM ведёт операционную работу, RabbitMQ передаёт события, а контур наблюдаемости помогает быстро находить проблемы.

## 4. Если бы в ближайшие полгода можно было сделать только три инициативы

При таком ограничении приоритет был бы таким:

1. **Переезд на Managed Kubernetes.** Это базовая инфраструктурная инициатива, без которой остальные улучшения будут либо дорогими, либо временными. Она сразу снижает риск простоя из-за одноинстансовой архитектуры, даёт масштабирование и упрощает деплой.

2. **Настроить метрики, алерты, трейсинг и дашборды.** Сейчас команда не видит, где именно теряются или застревают заказы. Без наблюдаемости любые исправления будут делаться почти вслепую. Эта инициатива даёт быстрый эффект: помогает находить узкие места и проверять, сработали ли следующие изменения.

3. **Перевести B2B-партнёров на Shop API.** Это самый полезный шаг с точки зрения бизнеса. Он убирает самый хрупкий сценарий, делает Shop API единой точкой входа и разгружает MES. После этого путь заказа становится короче и надёжнее.

Почему не остальные:

- **Изоляция MES как калькулятора** важна, но логически идёт сразу после перевода B2B на Shop API и заметно упрощается, когда первые три шага уже сделаны. Калькулятор можно будет удобно распараллелить в облаке на разных подах и снять bottleneck с системы расчетов.
- **Автоматизация E2E-тестов** полезна, но не снимает главные архитектурные риски быстрее, чем первые три пункта.
- **Исследование тормозов и разделения Shop DB** имеет смысл после появления метрик. Иначе мы снова будем гадать.
